# M5Stack CoreS3 桌面 AI 机器人助理项目整合梳理

## 1. 项目定位

本项目要做的不是单一的 CoreS3 玩具 Demo，也不是单纯的大模型 API 转发服务，而是一个由 **M5Stack CoreS3 桌面机器人终端** 和 **DeepNexus AI Gateway** 组成的桌面 AI 助理系统。

最终形态可以理解为：

```text
CoreS3 负责身体与交互
AI Gateway 负责智能与任务执行
```

CoreS3 端负责：

- 屏幕表情与状态动画
- 触摸交互
- Wi-Fi 连接
- 录音
- 语音播放
- 设备状态上报
- 接收后端命令
- 后续扩展摄像头、舵机、灯光、传感器

AI Gateway 端负责：

- 设备注册、心跳、状态管理
- STT 语音识别
- LLM 对话与推理
- TTS 语音合成
- Prompt 与人格管理
- Agent Runtime
- 提醒、番茄钟、记忆
- 模型路由、降级、成本记录
- 日志、控制台、运维能力

项目主线是先做一个可靠的半双工桌面 AI 机器人，再逐步升级成具备记忆、任务执行、视觉、动作和运营后台的产品化机器人。

## 3. 核心边界

### 3.1 CoreS3 边界

CoreS3 不直接理解模型、Prompt、Agent、记忆和成本。它只关心设备侧动作：

- 我当前是什么状态
- 我要不要录音
- 我要把音频上传到哪里
- 后端返回了什么文字、音频、表情、动作
- 我要播放什么声音
- 我要显示什么表情
- 我要上报什么事件

推荐 CoreS3 第一版状态机：

```text
BOOT
WIFI_CONNECTING
REGISTERING
IDLE
LISTENING
UPLOADING
THINKING
SPEAKING
SLEEPING
ERROR_STATE
```

### 3.2 AI Gateway 边界

AI Gateway 是系统智能中枢，不只是 `/ai/chat`：

- 统一设备入口
- 多模型路由
- STT / LLM / TTS 编排
- Prompt 版本管理
- Agent 任务执行
- Memory 读写
- Reminder 和 Pomodoro
- 统一返回协议
- 请求日志和成本追踪
- 后续控制台

## 4. MVP 请求链路

第一版建议做半双工语音闭环：

```text
用户点击 CoreS3 屏幕
↓
CoreS3 进入 LISTENING
↓
固定录音 5 秒
↓
上传到 POST /audio/chat
↓
Gateway 保存音频
↓
STT 转文字
↓
Intent Router 判断普通聊天或提醒任务
↓
Memory Service 加载上下文
↓
Model Router 选择模型
↓
Agent Runtime 生成结果
↓
TTS 生成回复音频
↓
返回 assistant_text / audio_url / face_state / device_actions
↓
CoreS3 播放音频并显示 SPEAKING 表情
↓
播放结束后回到 IDLE
```

## 6. 模块拆解

### 6.1 Firmware 模块

| 模块 | 职责 | 第一版目标 |
| --- | --- | --- |
| AppController | 总状态机与主循环 | 状态可切换，异常能恢复 |
| WiFiManager | Wi-Fi 连接与重连 | 能连上局域网 |
| DeviceClient | 调用 Gateway API | 注册、心跳、状态上报 |
| FaceEngine | 表情绘制 | IDLE/LISTENING/THINKING/SPEAKING/ERROR |
| TouchManager | 触摸事件 | 单击录音、双击打断、长按睡眠 |
| MicRecorder | 麦克风录音 | 固定 5 秒 WAV/PCM |
| SpeakerPlayer | 下载并播放音频 | 播放后端返回音频 |
| CommandHandler | 处理后端命令 | set_face/play_audio/reminder |

### 6.2 Gateway 模块

| 模块 | 职责 | 第一版目标 |
| --- | --- | --- |
| Device Gateway | 设备注册、心跳、命令队列 | CoreS3 可在线 |
| Audio Gateway | 音频上传、保存、返回音频 URL | `/audio/chat` 跑通 |
| STT Service | 语音转文字 | 先抽象接口，再接一个 Provider |
| LLM Service | 对话生成 | 支持一个主模型和兜底回复 |
| TTS Service | 文本转语音 | 生成 CoreS3 可播放音频 |
| Prompt Service | Prompt 加载 | `robot_default_v1` |
| Agent Runtime | 意图识别和任务执行 | ChatAgent + ReminderAgent |
| Memory Service | 用户偏好和项目记忆 | SQLite 简单记忆 |
| Observability | 请求日志、耗时、错误 | request_id 全链路 |

| 表 | 用途 |
| --- | --- |
| device | 设备注册信息 |
| device_status_log | 电量、RSSI、状态、固件版本 |

| device_event | 设备上报事件 |
| audio_asset | 用户录音和 TTS 结果 |
| conversation | 对话主记录 |

## 10. 分阶段路线

### Phase 0：工程初始化

目标：把项目变成可持续开发的工程。

交付：

- `firmware/` PlatformIO 工程
- `gateway/` FastAPI 工程
- `.env.example`
- `config.yaml`
- SQLite 初始化脚本
- `/health`
- 统一返回格式
- request_id 日志

### Phase 1：CoreS3 生命感 Demo

目标：不接 AI，先让机器人像一个可交互设备。

交付：

- 状态机
- 表情系统
- 触摸交互
- Wi-Fi 连接
- 断网错误表情

验收：

```text
开机 → 连 Wi-Fi → 待机脸 → 点击 → 聆听脸 → 思考脸 → 说话脸 → 待机脸
```

### Phase 2：Gateway 设备闭环

目标：CoreS3 能连上后端。

交付：

- `/device/register`
- `/device/heartbeat`
- `/device/status`
- `/device/{device_id}/commands`
- `/device/{device_id}/event`

验收：

```text
后端能看到 CoreS3 在线，并能下发简单命令
```

### Phase 3：语音对话闭环

目标：完成 MVP 最关键链路。

交付：

- CoreS3 录音
- multipart 上传
- STT Service
- LLM Service
- TTS Service
- 音频下载播放
- 对话日志

验收：

```text
点击 CoreS3 → 说话 → 上传 → AI 回复 → CoreS3 播放
```

### Phase 4：提醒、番茄钟、记忆

目标：从会聊天升级为能做事。

交付：

- ReminderAgent
- Pomodoro
- Memory Service
- 到点命令下发
- 提醒音频生成

验收：

```text
用户说“30 分钟后提醒我喝水”，机器人能创建任务并到点主动提醒
```

### Phase 5：人格、Prompt、本体论

目标：让机器人回复稳定、行为边界清晰。

交付：

- Prompt Center
- `robot_default_v1`
- `reminder_agent_v1`
- `memory_agent_v1`
- AgentProfile
- 轻量 Ontology

### Phase 6：控制台与运营能力

目标：可调试、可复盘、可运营。

交付：

- 设备列表
- 对话日志
- 音频资产回放
- 提醒任务管理
- 记忆管理
- Prompt 管理
- 请求日志
- 成本和错误统计

### Phase 7：语音唤醒与准全双工

目标：增强交互体验。

推荐顺序：

```text
触摸打断
↓
Gateway 侧唤醒词实验
↓
CoreS3 本地 WakeNet
↓
准全双工 / Barge-in
```

### Phase 8：视觉、动作、产品化

目标：从桌面 AI 终端升级为机器人产品。

交付方向：

- 摄像头拍照和视觉理解
- 舵机动作
- RGB LED / ToF 扩展
- OTA
- 用户账号和设备绑定
- 多设备管理
- 安全策略
- 商业化后台

## 14. 关键风险

| 风险 | 说明 | 建议 |
| --- | --- | --- |
| CoreS3 音频能力不稳定 | 录音、播放、网络上传同时做容易出问题 | 先固定 5 秒半双工，不急着全双工 |
| STT/TTS Provider 选择过早复杂化 | 一开始接太多模型会拖慢 MVP | 先抽象接口，只接一个可用 Provider |
| Prompt/Agent 过度设计 | 早期做太复杂会无法验证硬件链路 | 先 ChatAgent + ReminderAgent |
| 控制台过早投入 | UI 会吞掉大量时间 | 先靠日志和 SQLite 管理 |
| Wi-Fi 和后端地址配置麻烦 | 硬编码会影响调试 | `config.h` + `.env` 分别管理 |
| 返回协议不统一 | 设备端解析会混乱 | 一开始就固定 `code/msg/request_id/data` |
